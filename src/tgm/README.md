### ממשק מנוע אחיד
כל מנוע מייצא אובייקט עם:
```js
{ key, label, icon, color, signalType, description, generateLeads(dateMs) }
```
`generateLeads(dateMs)` מחזיר מערך לידים:
```js
{ symbol, source, signalType, timestamp, reason, entry, meta }
```
(`symbol/source/signalType/timestamp/reason` = הליבה לפי המפרט; `entry` = מחיר הסיגנל
באותו יום, `meta` = נתוני עזר. כדי להוסיף מנוע — כותבים קובץ חדש ומוסיפים ל-`engines/index.js`.)

## כלל הכניסה/יציאה (Daily Evaluator)

לכל ליד שנוצר (`evaluator.js`):

- **כניסה (entry):** מחיר הסיגנל ביום הליד = מחיר הפתיחה (`open`) של אותו יום.
- **יציאה (exit):** take-profit **+8%** / stop-loss **−4%**, מה שנגע קודם (מול ה-`high`/`low`
  התוך-יומיים). אם אף יעד לא נגע עד **סגירת היום** → יציאה במחיר הסגירה (`close`).
- **הכרעת סדר** כשגם TP וגם SL נגעו באותו נר (אין נתוני מסלול תוך-יומי): הסדר נגזר מכיוון
  הסגירה — נסגר מעל הכניסה → מניחים TP ראשון (win); נסגר מתחת → SL ראשון (loss). אומדן ניטרלי
  וריאליסטי יותר מ"SL תמיד ראשון".
- כל הלידים הם **LONG**.
- לכל ליד מחושב אחוז רווח/הפסד (`pnlPct`) וסטטוס, ונשמר ב-`localStorage`
  (מפתח `tgm_engine_leads_v1`) כרשומה מובנית.

**סטטוסים:** `win` (הצליח) · `loss` (נכשל) · `error` (לא הוכרע — לא נמשך מחיר יציאה).

לכל ליד נכתב **לוג ברור** לקונסול: `entry`, `exit`, `pnl`, `status`, סיבת יציאה/שגיאה.

## תיקון הבאג הקריטי — אחוז הצלחה שקרי (100%)

**הבעיה:** הדשבורד הציג 100% הצלחה בזמן שלידים סומנו "שגיאה" — סתירה שנבעה מחישוב שגוי.

**התיקון (`stats.js`, וגם בטבלת הסורק הקיים `engine/tgmStats.js`):**
1. ליד בסטטוס `error` **אינו** נספר כהצלחה ו**אינו** נכלל במכנה של ה-win-rate.
   נספר בנפרד כ"שגיאה / לא הוכרע".
2. `winRate = (מנצחים) / (טריידים שהוכרעו תקין)` — `win`+`loss` בלבד.
3. אם מספר הטריידים שהוכרעו = 0 → מציגים **"אין מספיק נתונים"**, לא 100%.
4. אם מספר הטריידים שהוכרעו < 10 (`MIN_SAMPLE_FOR_RATE`) → **"מדגם קטן מדי"**, לא מציגים אחוז
   (כך מדגם זעיר אינו מציג 100% מטעה).
5. **מקור השגיאות:** נחקר — לידים נכשלים כשמשיכת מחיר היציאה נכשלת. בשכבת ה-MOCK זה מדומה
   באמצעות `getExitBar`, שמחזיר `null` עבור מניות "דקות נתונים" (`thinData`) בחלק מהימים —
   בדיוק התרחיש שמ"חור" ב-API אמיתי. **TODO:** מול ספק אמיתי לטפל ב-rate-limit / סימבול חסר /
   יום ללא מסחר, ולא לספור ככישלון.

## סטטיסטיקה חודשית + דירוג מנועים (`stats.js`)

לכל מנוע בנפרד וגם לכולם יחד, פר-חודש:
- **Win Rate** (רק טריידים שהוכרעו תקין)
- **תשואה ממוצעת לליד**
- **מספר לידים שנוצרו** / הוכרעו / נכשלו / בשגיאה
- **Profit Factor** = סך רווחים / |סך הפסדים| (`∞` כשאין הפסדים)
- **Max Drawdown** (על עקומת ההון המצטברת של הטריידים שהוכרעו)

**דירוג מנועים:** מהטוב לגרוע לפי Win Rate (ואז Profit Factor, ואז תשואה ממוצעת).
דירוג רשמי רק למנוע עם ≥ 10 טריידים שהוכרעו.

## ה-UI

מתג בראש עמוד TGM עובר בין **🧪 מעבדת מנועים** (החדש) ל-**📡 סורק טלגרם** (הקיים).
מעבדת המנועים כוללת:
- **דשבורד:** שלושה מספרים נפרדים — הצליחו / נכשלו / בשגיאה — + KPIs חודשיים + טבלת דירוג מנועים.
- **טאב לכל מנוע:** הלידים היומיים שלו, צבועים — ירוק=הצליח, אדום=נכשל, צהוב/זהב=שגיאה.
- בורר חודש לסטטיסטיקה החודשית.
- אם מדגם < 10 → "מדגם קטן מדי" (לא מוצג אחוז).

בטעינה ראשונה, אם אין נתונים, המערכת זורעת אוטומטית היסטוריה של 30 ימי מסחר
(`store.seedHistory`) כדי שיהיה מדגם להצגה. כפתורים: "הרץ סבב יומי", "זרע היסטוריה חודשית", "נקה".
